# Code Review Checklist

本文件包含代码审查的详细检查规则，分为**通用规则**（前后端都适用）、**前端专属规则**和**后端专属规则**。根据项目配置文件 `.skills/code-review/.review-config.md` 中启用的检查类别，逐一应用以下规则。

---

## 一、通用规则（前后端都适用）

### 1.1 代码质量

#### 命名规范
- [ ] 变量/函数名是否清晰表达意图（避免 `data`、`temp`、`handle` 等模糊命名）
- [ ] 类名是否使用 PascalCase
- [ ] 函数/方法名是否使用 snake_case（Python/Ruby）或 camelCase（Java/JS/Go）
- [ ] 常量名是否使用 UPPER_SNAKE_CASE
- [ ] 文件名是否与导出的类/模块名一致

#### 类型安全
- [ ] 函数参数和返回值是否有类型标注（TypeScript/Python type hints/Go 类型）
- [ ] 是否避免使用 `any`/`interface{}`/`Object` 等宽泛类型
- [ ] API 请求/响应是否定义了数据模型（DTO/Schema/Interface）
- [ ] 数据库模型是否有完整的类型定义

#### 代码结构
- [ ] 函数/类是否职责单一（单一职责原则）
- [ ] 是否存在重复代码（可提取为公共函数/工具类）
- [ ] 函数长度是否合理（建议不超过 50 行）
- [ ] 类/组件是否过度复杂（建议拆分）
- [ ] 条件逻辑是否过于嵌套（建议提前 return 或提取函数）
- [ ] 是否遵循项目的分层架构（Controller → Service → Repository）

#### 导入/依赖
- [ ] 是否有未使用的导入
- [ ] 是否有未使用的变量/函数
- [ ] 导入顺序是否合理（标准库 → 第三方库 → 内部模块）
- [ ] 是否存在循环依赖
- [ ] 依赖注入是否合理（避免过度耦合）

---

### 1.2 安全性

#### 注入攻击防护
- [ ] 数据库查询是否使用参数化查询/ORM（避免字符串拼接 SQL）
- [ ] 用户输入在展示前是否经过转义（防止 XSS）
- [ ] 动态查询条件是否使用了安全的构建方式
- [ ] 用户输入是否进行了类型和长度校验

#### 认证与授权
- [ ] API 是否有认证机制（JWT/OAuth/Session）
- [ ] 敏感操作是否进行了权限校验
- [ ] Token 是否设置了合理的过期时间
- [ ] 是否防止了权限提升攻击

#### 敏感数据保护
- [ ] 密码是否使用安全的哈希算法（bcrypt/argon2，非 MD5/SHA1）
- [ ] 敏感配置（数据库密码、API Key）是否从环境变量/配置中心读取
- [ ] 日志中是否避免了输出敏感信息（密码、Token、身份证号）
- [ ] API 响应中是否避免了返回敏感字段
- [ ] 是否在注释中遗留了敏感信息

#### 其他安全
- [ ] 是否限制了 API 请求频率（Rate Limiting）
- [ ] 是否使用了 HTTPS
- [ ] CORS 配置是否合理（非 `*`）
- [ ] 是否避免了信息泄露（错误堆栈、版本号）
- [ ] 是否有 CSRF 防护措施

---

### 1.3 错误处理

#### 异常捕获
- [ ] 是否有全局异常处理器
- [ ] 是否避免了捕获所有异常（`catch (Exception e)`）
- [ ] 是否区分了业务异常和系统异常
- [ ] 异常是否被正确传播（避免吞掉异常）

#### 日志记录
- [ ] 关键操作是否记录了日志
- [ ] 日志级别是否合理（DEBUG/INFO/WARN/ERROR）
- [ ] 日志内容是否包含足够的上下文（请求 ID、用户 ID、参数）
- [ ] 是否避免了在日志中输出敏感信息
- [ ] 日志格式是否统一（便于日志分析系统解析）

#### 降级与重试
- [ ] 外部服务调用是否有降级策略
- [ ] 是否有熔断机制（Circuit Breaker）
- [ ] 重试是否使用了指数退避
- [ ] 重试是否限制了最大次数

---

## 二、前端专属规则

### 2.1 性能

#### React 特定
- [ ] 是否避免不必要的重渲染（合理使用 `React.memo`、`useMemo`、`useCallback`）
- [ ] 是否在渲染函数中创建了新的对象/数组/函数（应提取到组件外或使用 memo）
- [ ] 列表渲染是否使用了稳定的 `key`（避免使用 index 作为 key）
- [ ] 是否避免了在 render 中进行昂贵的计算
- [ ] Context 使用是否合理（避免过度使用导致不必要的重渲染）

#### 资源加载
- [ ] 图片是否使用了懒加载（`loading="lazy"` 或 Intersection Observer）
- [ ] 大型组件/页面是否使用了代码分割（`React.lazy`、动态 `import()`）
- [ ] 是否避免了阻塞主线程的同步操作
- [ ] 第三方库是否按需导入（避免全量导入）

#### 内存管理
- [ ] `useEffect` 中的定时器/事件监听器/订阅是否在 cleanup 中清理
- [ ] 是否存在闭包导致的内存泄漏
- [ ] 大数组/对象是否及时释放引用

---

### 2.2 可访问性

#### 语义化 HTML
- [ ] 是否使用了语义化标签（`<header>`、`<nav>`、`<main>`、`<article>`、`<section>`、`<footer>`）
- [ ] 标题层级是否正确（`<h1>` → `<h2>` → `<h3>`，不跳级）
- [ ] 链接文本是否有意义（避免"点击这里"）

#### ARIA 属性
- [ ] 自定义组件是否有正确的 ARIA 角色（`role`）
- [ ] 交互元素是否有 `aria-label` 或 `aria-labelledby`
- [ ] 模态框/弹窗是否使用了 `aria-modal` 和焦点管理

#### 键盘导航
- [ ] 所有交互元素是否可通过键盘访问（Tab 键）
- [ ] 是否有合理的焦点顺序
- [ ] 自定义组件是否支持键盘操作（Enter/Space/Escape/方向键）

#### 视觉可访问性
- [ ] 文本颜色对比度是否足够（WCAG AA: 4.5:1 正文，3:1 大文本）
- [ ] 是否不仅依赖颜色传达信息（辅以图标/文字）
- [ ] 焦点指示器是否可见
- [ ] 图片是否有 `alt` 属性（装饰性图片使用 `alt=""`）

#### 表单
- [ ] 每个表单元素是否关联了 `<label>`（`htmlFor` / `aria-label`）
- [ ] 必填字段是否标注了 `required`
- [ ] 错误信息是否与表单元素关联（`aria-describedby`）

---

### 2.3 样式

#### CSS 规范
- [ ] 类名是否遵循项目约定的命名规范（BEM / camelCase / kebab-case）
- [ ] 是否避免了过深的选择器嵌套（建议不超过 3 层）
- [ ] 是否避免了 `!important`（除非有充分理由）
- [ ] 是否避免了内联样式（除非动态计算）

#### 响应式设计
- [ ] 是否考虑了移动端适配
- [ ] 是否使用了相对单位（rem/em/vw/vh）而非固定像素
- [ ] 图片和媒体元素是否自适应

#### 设计系统一致性
- [ ] 颜色是否使用了设计 token/变量（而非硬编码色值）
- [ ] 间距是否使用了统一的间距系统
- [ ] 字体大小/字重是否使用了设计系统的值

---

## 三、后端专属规则

### 3.1 性能

#### 数据库查询
- [ ] 是否存在 N+1 查询问题（应使用 JOIN 或预加载）
- [ ] 大数据量查询是否使用了分页
- [ ] 是否避免了 `SELECT *`（只查询需要的字段）
- [ ] 是否在合适的字段上建立了索引
- [ ] 是否避免了在循环中执行数据库查询
- [ ] 批量操作是否使用了批量插入/更新

#### 缓存使用
- [ ] 频繁读取的数据是否使用了缓存（Redis/Memcached）
- [ ] 缓存是否设置了合理的过期时间
- [ ] 是否处理了缓存穿透/击穿/雪崩问题
- [ ] 缓存更新策略是否合理

#### 并发处理
- [ ] 共享资源访问是否使用了锁/事务
- [ ] 是否避免了死锁（锁顺序一致、超时机制）
- [ ] 是否使用了连接池（数据库/HTTP/Redis）
- [ ] 异步任务是否使用了消息队列

#### 内存管理
- [ ] 大数据处理是否使用了流式处理（避免一次性加载到内存）
- [ ] 是否及时释放了不再使用的资源（文件句柄、连接）
- [ ] 是否避免了内存泄漏

---

### 3.2 API 设计

#### RESTful 规范
- [ ] URL 是否使用名词复数（`/users` 而非 `/getUser`）
- [ ] HTTP 方法是否正确（GET 查询/POST 创建/PUT 更新/DELETE 删除）
- [ ] 状态码是否正确（200 成功/201 创建/400 参数错误/401 未认证/403 无权限/404 不存在/500 服务器错误）
- [ ] 是否支持了合适的查询参数（分页/排序/过滤）

#### 请求/响应设计
- [ ] 请求体是否使用了 JSON 格式
- [ ] 响应格式是否统一（包含 code/message/data）
- [ ] 大数据响应是否使用了压缩（gzip）

#### 版本控制与兼容性
- [ ] API 是否有版本号（`/v1/users`）
- [ ] 新增字段是否向后兼容
- [ ] 废弃字段是否有标记和迁移方案
- [ ] 是否避免了破坏性变更

#### 幂等性
- [ ] POST/DELETE 操作是否考虑了幂等性
- [ ] 是否使用了幂等键（Idempotency-Key）

---

### 3.3 数据库

#### 索引优化
- [ ] 查询条件字段是否建立了索引
- [ ] 是否避免了过多索引（影响写入性能）
- [ ] 复合索引字段顺序是否合理

#### 事务管理
- [ ] 关键业务操作是否使用了事务
- [ ] 事务范围是否最小化（避免长事务）
- [ ] 是否避免了在事务中进行外部调用
- [ ] 是否正确处理了事务回滚

#### 数据一致性
- [ ] 是否使用了合适的隔离级别
- [ ] 是否防止了并发更新问题（乐观锁/悲观锁）
- [ ] 唯一约束是否正确设置

#### 数据迁移
- [ ] 数据库变更是否有迁移脚本
- [ ] 迁移是否可回滚
- [ ] 大数据量迁移是否分批执行

#### SQL 质量
- [ ] 是否避免了复杂的子查询（可用 JOIN 替代）
- [ ] 是否避免了在 WHERE 中使用函数（导致索引失效）
- [ ] 是否使用了 EXPLAIN 分析慢查询

---

## 四、严重级别判定指南

| 级别 | 判定标准 | 示例 |
|------|----------|------|
| **P0（紧急）** | 安全漏洞、数据丢失、生产崩溃、认证绕过 | SQL 注入、硬编码密码、未授权访问、XSS 漏洞 |
| **P1（重要）** | API 契约错误、用户可见回归、数据库死锁、内存泄漏 | 接口返回错误格式、N+1 查询导致超时、页面白屏 |
| **P2（建议）** | 可维护性问题、性能隐患、缺少测试 | 缺少错误处理、重复代码、缺少索引、缺少 key 属性 |
| **P3（优化）** | 代码风格、命名优化、注释补充 | 命名不够清晰、缺少注释、样式可优化 |
