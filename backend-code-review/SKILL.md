# Backend Code Review

对后端代码进行结构化审查。首次使用时自动检测项目技术栈并生成审查配置模板，后续调用基于模板进行审查。

## Intent

当用户要求审查、检查、分析后端代码时触发此技能。支持以下文件类型：
- `.py`, `.java`, `.go`, `.ts`, `.js`, `.rb`, `.php`, `.rs`, `.kt`, `.scala`
- `.sql`, `.graphql`
- `Dockerfile`, `docker-compose.yml`
- `.yaml`, `.yml`, `.toml`, `.json`（配置文件）

## Review Modes

根据用户意图自动判断使用哪种模式。如果用户没有明确指定，**默认使用模式 B（Git 变更审查）**。

### Mode A: 项目全量审查

**触发条件**：用户明确说"审查整个项目"、"全量审查"、"检查所有代码"
**审查范围**：项目中所有后端文件（排除配置中的排除路径）
**适用场景**：项目初始化后的首次审查、定期代码健康检查

### Mode B: Git 变更审查（默认）

**触发条件**：用户说"审查我的修改"、"检查这次改动"、"提交前审查"、"review 一下"、或任何未明确指定范围的审查请求
**审查范围**：仅本次 git 变更涉及的文件和代码行
**适用场景**：日常开发中的小改动、提交前的快速检查

支持以下 git 范围（按优先级自动判断）：

| 优先级 | 范围 | Git 命令 | 说明 |
|--------|------|----------|------|
| 1 | Staged 变更 | `git diff --cached` | 已 `git add` 但未 commit 的变更 |
| 2 | Unstaged 变更 | `git diff` | 已修改但未 `git add` 的变更 |
| 3 | 最近一次 commit | `git diff HEAD~1 HEAD` | 最后一次提交的变更 |
| 4 | 当前分支 vs 主分支 | `git diff main...HEAD` 或 `git diff master...HEAD` | 当前分支所有变更 |

如果同时存在 staged 和 unstaged 变更，**优先审查 staged 变更**，并提示用户还有 unstaged 变更未审查。

### Mode C: 文件定向审查

**触发条件**：用户指定了具体文件或目录路径
**审查范围**：用户指定的文件或目录
**适用场景**：审查特定模块、API、服务

## First-Time Setup: 技术栈检测与模板初始化

### 检测流程

首次调用时，检查项目根目录下是否存在配置文件 `.skills/backend-code-review/.review-config.md`。

**如果不存在**，执行以下初始化流程：

1. **扫描项目根目录**，检测以下信息：

|--------|----------|
| 语言 | 检测 `.py`/`.java`/`.go`/`.ts`/`.js`/`.rb`/`.php`/`.rs` 等文件占比 |
| 框架 | Python: Django/Flask/FastAPI; Java: Spring Boot; Node: Express/Nest.js; Go: Gin/Echo; Ruby: Rails; PHP: Laravel |
| ORM/数据库 | SQLAlchemy/Django ORM/Prisma/TypeORM/GORM/ActiveRecord; MySQL/PostgreSQL/MongoDB/Redis |
| 消息队列 | RabbitMQ/Kafka/Redis Queue/Celery/Bull |
| 缓存 | Redis/Memcached/本地缓存 |
| 测试框架 | pytest/JUnit/Jest/Mocha/RSpec/Go testing |
| API 类型 | REST/GraphQL/gRPC |
| 部署方式 | Docker/Kubernetes/Serverless/传统部署 |
| 代码规范 | ESLint/Pylint/Black/Go fmt/RuboCop/PHP-CS-Fixer |
| 包管理器 | pip/poetry/maven/gradle/npm/go modules/gem/composer |

## Review Process

### Mode A: 项目全量审查

1. 扫描项目中所有后端文件（排除配置中的排除路径）
2. 按文件逐一检查
3. 生成全量报告

### Mode B: Git 变更审查（重点）

#### Step 1: 获取变更范围

```bash
# 1. 先检查是否有 staged 变更
git diff --cached --stat
git diff --cached

# 2. 如果没有 staged，检查 unstaged 变更
git diff --stat
git diff

# 3. 如果都没有，检查最近一次 commit
git diff HEAD~1 HEAD --stat
git diff HEAD~1 HEAD

# 4. 如果都没有，检查当前分支 vs 主分支
git diff main...HEAD --stat  # 或 master...HEAD
git diff main...HEAD
```

#### Step 2: 过滤后端文件

从变更文件列表中，只保留后端相关文件（`.py`, `.java`, `.go`, `.ts`, `.js`, `.rb`, `.php`, `.rs`, `.kt`, `.scala`, `.sql`, `.graphql`, `Dockerfile`, `.yaml`, `.yml`, `.toml`, `.json`），排除配置中的排除路径。

如果没有后端文件变更，提示用户"本次变更不涉及后端文件"。

#### Step 3: 生成业务级变更概述

在分析 diff 的基础上，用**自然语言**总结本次变更的业务含义。这是让开发者一眼看懂"这次改了什么"的关键。

**生成方法**：
1. 读取所有变更文件的 diff 内容
2. 理解每个文件变更的业务意图（新增/修改/删除了什么功能）
3. 用简洁的中文概述每条业务变更

**概述要求**：
- 用**业务语言**而非技术语言描述（"新增用户注册接口"而非"添加了 50 行 Python 代码"）
- 每条概述控制在一句话内
- 按影响范围从大到小排列
- 标注变更类型：新增 / 修改 / 移除 / 重构 / 修复

**示例**：
```
- [新增] 用户注册接口，支持邮箱验证码校验
- [修改] 订单创建逻辑，增加库存预占机制防止超卖
- [修复] 修复了并发场景下优惠券重复使用的问题
- [重构] 将支付回调处理从 Controller 迁移到 Service 层
- [移除] 废弃的 v1 版本用户查询接口
```

#### Step 4: 分析变更摘要

生成技术层面的变更摘要，包括：
- 变更文件数量
- 新增/修改/删除的文件列表
- 变更行数统计（+N / -N）

#### Step 5: 逐文件审查变更

对每个变更文件：

1. **读取 diff 内容**：只关注变更的代码行（`+` 和 `-` 行）
2. **读取上下文**：读取变更行周围的相关代码（前后各 20 行），理解模块结构、依赖关系、数据流
3. **检查变更部分**：根据配置中启用的检查类别，只审查变更的代码，不审查未变更的代码
4. **检查变更引入的问题**：
   - 新增代码是否引入了 bug
   - 修改代码是否破坏了原有逻辑
   - 删除代码是否影响了其他依赖
   - 变更是否与项目规范一致

#### Step 6: 生成变更级报告

### Mode C: 文件定向审查

1. 读取用户指定的文件
2. 按检查类别全量审查
3. 生成文件级报告

## Checklist

详细审查规则见 [references/checklist.md](references/checklist.md)。

按配置文件中启用的检查类别，逐一检查：

- **代码质量**：命名、类型安全、职责单一、重复代码、导入导出
- **安全性**：SQL 注入、XSS、认证授权、敏感数据保护、CSRF
- **性能**：数据库查询优化、缓存使用、并发处理、内存管理
- **API 设计**：RESTful 规范、错误响应、版本控制、幂等性
- **错误处理**：异常捕获、日志记录、降级策略、重试机制
- **数据库**：索引使用、事务管理、数据一致性、N+1 查询

## Rules

1. **Git 变更审查只审查变更的代码**，不审查未变更的部分，保持报告简洁
2. 但需要**读取上下文代码**来理解变更的影响范围
3. 审查结果应具体、可操作，不要给出模糊的建议
4. 每个问题都要指出具体的文件和行号，并展示相关的 diff 内容
5. 如果问题数量超过 10 个，先输出"10+ 个问题"，然后只展示前 10 个最严重的问题
6. 不要包含主观偏好，只关注客观的代码质量
7. 审查完成后，询问用户是否需要帮助修复问题
8. 如果用户没有明确指定审查模式，**默认使用 Git 变更审查**
9. 如果没有发现问题，使用 Template B-B 输出简洁的"无问题"报告
10. 对于数据库相关变更，特别注意索引、事务、数据迁移的影响
11. 对于 API 变更，检查向后兼容性和版本控制
