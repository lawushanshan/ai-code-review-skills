# Backend Code Review Checklist

本文件包含后端代码审查的详细检查规则。根据项目配置文件 `.skills/backend-code-review/.review-config.md` 中启用的检查类别，逐一应用以下规则。

---

## 1. 代码质量 (Code Quality)

### 命名规范
- [ ] 变量/函数名是否清晰表达意图（避免 `data`、`temp`、`handle` 等模糊命名）
- [ ] 类名是否使用 PascalCase（Java/Python/Go）
- [ ] 函数/方法名是否使用 snake_case（Python/Ruby）或 camelCase（Java/JS/Go）
- [ ] 常量名是否使用 UPPER_SNAKE_CASE
- [ ] 文件名是否与导出的类/模块名一致
- [ ] 数据库字段命名是否遵循项目约定（snake_case）

### 类型安全
- [ ] 函数参数和返回值是否有类型标注（TypeScript/Python type hints/Go 类型）
- [ ] 是否避免使用 `any`/`interface{}`/`Object` 等宽泛类型
- [ ] API 请求/响应是否定义了数据模型（DTO/Schema）
- [ ] 数据库模型是否有完整的类型定义
- [ ] 泛型使用是否合理（避免过度嵌套）

### 代码结构
- [ ] 函数/类是否职责单一（单一职责原则）
- [ ] 是否存在重复代码（可提取为公共函数/工具类）
- [ ] 函数长度是否合理（建议不超过 50 行）
- [ ] 类是否过度复杂（建议拆分为多个类）
- [ ] 条件逻辑是否过于嵌套（建议提前 return 或提取函数）
- [ ] 是否遵循项目的分层架构（Controller → Service → Repository）

### 导入/依赖
- [ ] 是否有未使用的导入
- [ ] 是否有未使用的变量/函数
- [ ] 导入顺序是否合理（标准库 → 第三方库 → 内部模块）
- [ ] 是否存在循环依赖
- [ ] 依赖注入是否合理（避免过度耦合）

---

## 2. 安全性 (Security)

### SQL 注入防护
- [ ] 数据库查询是否使用参数化查询/ORM（避免字符串拼接 SQL）
- [ ] 用户输入在拼接到 SQL 前是否经过转义
- [ ] 动态查询条件是否使用了安全的构建方式
- [ ] 存储过程调用是否安全

### 认证与授权
- [ ] API 是否有认证机制（JWT/OAuth/Session）
- [ ] 敏感操作是否进行了权限校验
- [ ] 是否实现了 RBAC/ABAC 权限模型
- [ ] Token 是否设置了合理的过期时间
- [ ] 是否防止了权限提升攻击

### 敏感数据保护
- [ ] 密码是否使用安全的哈希算法（bcrypt/argon2，非 MD5/SHA1）
- [ ] 敏感配置（数据库密码、API Key）是否从环境变量/配置中心读取
- [ ] 日志中是否避免了输出敏感信息（密码、Token、身份证号）
- [ ] API 响应中是否避免了返回敏感字段
- [ ] 是否在注释中遗留了敏感信息

### 输入校验
- [ ] 用户输入是否进行了类型和长度校验
- [ ] 文件上传是否校验了类型、大小和内容
- [ ] 是否防止了路径遍历攻击（`../`）
- [ ] 是否防止了 SSRF（服务端请求伪造）
- [ ] 是否有 CSRF 防护措施

### 其他安全
- [ ] 是否限制了 API 请求频率（Rate Limiting）
- [ ] 是否防止了重放攻击
- [ ] 是否使用了 HTTPS
- [ ] CORS 配置是否合理（非 `*`）
- [ ] 是否避免了信息泄露（错误堆栈、版本号）

---

## 3. 性能 (Performance)

### 数据库查询
- [ ] 是否存在 N+1 查询问题（应使用 JOIN 或预加载）
- [ ] 大数据量查询是否使用了分页
- [ ] 是否避免了 `SELECT *`（只查询需要的字段）
- [ ] 是否在合适的字段上建立了索引
- [ ] 是否避免了在循环中执行数据库查询
- [ ] 批量操作是否使用了批量插入/更新

### 缓存使用
- [ ] 频繁读取的数据是否使用了缓存（Redis/Memcached）
- [ ] 缓存是否设置了合理的过期时间
- [ ] 是否处理了缓存穿透/击穿/雪崩问题
- [ ] 缓存更新策略是否合理（Cache Aside/Write Through/Write Behind）
- [ ] 是否避免了缓存与数据库的数据不一致

### 并发处理
- [ ] 共享资源访问是否使用了锁/事务
- [ ] 是否避免了死锁（锁顺序一致、超时机制）
- [ ] 是否使用了连接池（数据库/HTTP/Redis）
- [ ] 异步任务是否使用了消息队列
- [ ] 是否避免了阻塞主线程的操作

### 内存管理
- [ ] 大数据处理是否使用了流式处理（避免一次性加载到内存）
- [ ] 是否及时释放了不再使用的资源（文件句柄、连接）
- [ ] 是否避免了内存泄漏（未关闭的资源、缓存无限增长）
- [ ] 字符串拼接是否使用了高效方式（StringBuilder/join）

### 网络与 I/O
- [ ] HTTP 请求是否设置了超时时间
- [ ] 是否使用了连接复用（Keep-Alive）
- [ ] 是否避免了同步阻塞 I/O（应使用异步/非阻塞）
- [ ] 文件操作是否使用了缓冲

---

## 4. API 设计 (API Design)

### RESTful 规范
- [ ] URL 是否使用名词复数（`/users` 而非 `/getUser`）
- [ ] HTTP 方法是否正确（GET 查询/POST 创建/PUT 更新/DELETE 删除）
- [ ] 状态码是否正确（200 成功/201 创建/400 参数错误/401 未认证/403 无权限/404 不存在/500 服务器错误）
- [ ] 是否支持了合适的查询参数（分页/排序/过滤）

### 请求/响应设计
- [ ] 请求体是否使用了 JSON 格式
- [ ] 响应格式是否统一（包含 code/message/data）
- [ ] 是否支持了合适的 Content-Type
- [ ] 大数据响应是否使用了压缩（gzip）

### 错误处理
- [ ] 错误响应是否包含有用的信息（错误码、错误描述）
- [ ] 是否避免了在错误响应中暴露敏感信息
- [ ] 参数校验失败是否返回了详细的错误信息
- [ ] 是否有全局异常处理器

### 版本控制与兼容性
- [ ] API 是否有版本号（`/v1/users`）
- [ ] 新增字段是否向后兼容
- [ ] 废弃字段是否有标记和迁移方案
- [ ] 是否避免了破坏性变更

### 幂等性
- [ ] POST/DELETE 操作是否考虑了幂等性
- [ ] 是否使用了幂等键（Idempotency-Key）
- [ ] 重试机制是否安全

---

## 5. 错误处理 (Error Handling)

### 异常捕获
- [ ] 是否有全局异常处理器
- [ ] 是否避免了捕获所有异常（`catch (Exception e)`）
- [ ] 是否区分了业务异常和系统异常
- [ ] 异常是否被正确传播（避免吞掉异常）

### 日志记录
- [ ] 关键操作是否记录了日志
- [ ] 日志级别是否合理（DEBUG/INFO/WARN/ERROR）
- [ ] 日志内容是否包含足够的上下文（请求 ID、用户 ID、参数）
- [ ] 是否避免了在日志中输出敏感信息
- [ ] 日志格式是否统一（便于日志分析系统解析）

### 降级与重试
- [ ] 外部服务调用是否有降级策略
- [ ] 是否有熔断机制（Circuit Breaker）
- [ ] 重试是否使用了指数退避
- [ ] 重试是否限制了最大次数

### 数据一致性
- [ ] 分布式操作是否考虑了最终一致性
- [ ] 是否有补偿机制（Saga 模式）
- [ ] 消息队列消费是否保证了幂等性

---

## 6. 数据库 (Database)

### 索引优化
- [ ] 查询条件字段是否建立了索引
- [ ] 是否避免了过多索引（影响写入性能）
- [ ] 复合索引字段顺序是否合理
- [ ] 是否使用了覆盖索引

### 事务管理
- [ ] 关键业务操作是否使用了事务
- [ ] 事务范围是否最小化（避免长事务）
- [ ] 是否避免了在事务中进行外部调用
- [ ] 是否正确处理了事务回滚

### 数据一致性
- [ ] 是否使用了合适的隔离级别
- [ ] 是否防止了并发更新问题（乐观锁/悲观锁）
- [ ] 唯一约束是否正确设置
- [ ] 外键约束是否合理

### 数据迁移
- [ ] 数据库变更是否有迁移脚本
- [ ] 迁移是否可回滚
- [ ] 大数据量迁移是否分批执行
- [ ] 是否避免了锁表操作

### SQL 质量
- [ ] 是否避免了复杂的子查询（可用 JOIN 替代）
- [ ] 是否避免了在 WHERE 中使用函数（导致索引失效）
- [ ] 是否避免了隐式类型转换
- [ ] 是否使用了 EXPLAIN 分析慢查询

---

## 严重级别判定指南

| 级别 | 判定标准 | 示例 |
|------|----------|------|
| **P0（紧急）** | 安全漏洞、数据丢失、生产崩溃、认证绕过 | SQL 注入、硬编码密码、未授权访问、数据泄露 |
| **P1（重要）** | API 契约错误、数据库死锁、内存泄漏、性能严重下降 | 接口返回错误格式、N+1 查询导致超时、连接泄漏 |
| **P2（建议）** | 可维护性问题、性能隐患、缺少测试、日志不完整 | 缺少错误处理、重复代码、缺少索引、日志不完整 |
| **P3（优化）** | 代码风格、命名优化、注释补充、微小改进 | 命名不够清晰、缺少注释、代码可简化 |
