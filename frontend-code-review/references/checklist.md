# Frontend Code Review Checklist

本文件包含前端代码审查的详细检查规则。根据项目配置文件 `.skills/frontend-code-review/.review-config.md` 中启用的检查类别，逐一应用以下规则。

---

## 1. 代码质量 (Code Quality)

### 命名规范
- [ ] 变量/函数名是否清晰表达意图（避免 `data`、`temp`、`handleClick` 等模糊命名）
- [ ] 组件名是否使用 PascalCase
- [ ] 函数/变量名是否使用 camelCase
- [ ] 常量名是否使用 UPPER_SNAKE_CASE
- [ ] 文件名是否与导出的组件/模块名一致

### 类型安全 (TypeScript)
- [ ] 函数参数和返回值是否有类型标注
- [ ] 是否避免使用 `any` 类型（使用 `unknown` 或具体类型替代）
- [ ] Props 是否定义了 interface/type
- [ ] API 响应数据是否定义了类型
- [ ] 泛型使用是否合理（避免过度嵌套）

### 代码结构
- [ ] 函数/组件是否职责单一（单一职责原则）
- [ ] 是否存在重复代码（可提取为公共函数/组件）
- [ ] 函数长度是否合理（建议不超过 50 行）
- [ ] 组件是否过度复杂（建议拆分为子组件）
- [ ] 条件逻辑是否过于嵌套（建议提前 return 或提取函数）

### 导入/导出
- [ ] 是否有未使用的导入
- [ ] 是否有未使用的变量/函数
- [ ] 导入顺序是否合理（外部库 → 内部模块 → 样式/资源）
- [ ] 导出是否合理（避免过度导出内部实现细节）

---

## 2. 性能 (Performance)

### React 特定
- [ ] 是否避免不必要的重渲染（合理使用 `React.memo`、`useMemo`、`useCallback`）
- [ ] 是否在渲染函数中创建了新的对象/数组/函数（应提取到组件外或使用 memo）
- [ ] 列表渲染是否使用了稳定的 `key`（避免使用 index 作为 key）
- [ ] 是否避免了在 render 中进行昂贵的计算
- [ ] Context 使用是否合理（避免过度使用导致不必要的重渲染）

### 资源加载
- [ ] 图片是否使用了懒加载（`loading="lazy"` 或 Intersection Observer）
- [ ] 大型组件/页面是否使用了代码分割（`React.lazy`、动态 `import()`）
- [ ] 是否避免了阻塞主线程的同步操作
- [ ] 第三方库是否按需导入（避免 `import { Button } from 'antd'` 全量导入）

### 内存管理
- [ ] `useEffect` 中的定时器/事件监听器/订阅是否在 cleanup 中清理
- [ ] 是否存在闭包导致的内存泄漏
- [ ] 大数组/对象是否及时释放引用
- [ ] 是否避免了在循环中创建闭包

### 网络请求
- [ ] 是否存在重复的 API 请求（应使用缓存或去重）
- [ ] 是否避免了瀑布式请求（应并行请求或合并接口）
- [ ] 大数据量是否使用了分页/虚拟滚动

---

## 3. 可访问性 (Accessibility)

### 语义化 HTML
- [ ] 是否使用了语义化标签（`<header>`、`<nav>`、`<main>`、`<article>`、`<section>`、`<footer>`）
- [ ] 标题层级是否正确（`<h1>` → `<h2>` → `<h3>`，不跳级）
- [ ] 列表是否使用了 `<ul>`/`<ol>`/`<li>`
- [ ] 链接文本是否有意义（避免"点击这里"）

### ARIA 属性
- [ ] 自定义组件是否有正确的 ARIA 角色（`role`）
- [ ] 交互元素是否有 `aria-label` 或 `aria-labelledby`
- [ ] 模态框/弹窗是否使用了 `aria-modal` 和焦点管理
- [ ] 加载状态是否有 `aria-live` 或 `aria-busy` 提示

### 键盘导航
- [ ] 所有交互元素是否可通过键盘访问（Tab 键）
- [ ] 是否有合理的焦点顺序
- [ ] 自定义组件是否支持键盘操作（Enter/Space/Escape/方向键）
- [ ] 是否避免了 `tabIndex` 大于 0 的值

### 视觉可访问性
- [ ] 文本颜色对比度是否足够（WCAG AA: 4.5:1 正文，3:1 大文本）
- [ ] 是否不仅依赖颜色传达信息（辅以图标/文字）
- [ ] 焦点指示器是否可见
- [ ] 图片是否有 `alt` 属性（装饰性图片使用 `alt=""`）

### 表单
- [ ] 每个表单元素是否关联了 `<label>`（`htmlFor` / `aria-label`）
- [ ] 必填字段是否标注了 `required`
- [ ] 错误信息是否与表单元素关联（`aria-describedby`）
- [ ] 输入类型是否正确（`email`、`tel`、`number` 等）

---

## 4. 安全性 (Security)

### XSS 防护
- [ ] 是否避免了 `dangerouslySetInnerHTML`（React）/ `v-html`（Vue）
- [ ] 如果必须使用，输入是否经过了 sanitize 处理
- [ ] 用户输入在展示前是否经过了转义
- [ ] URL 参数是否经过了验证和编码

### 敏感信息
- [ ] API 密钥/Token 是否硬编码在客户端代码中
- [ ] 是否在日志中输出了敏感信息（密码、Token、个人信息）
- [ ] localStorage/sessionStorage 中是否存储了敏感数据
- [ ] 是否在注释中遗留了敏感信息

### 输入校验
- [ ] 用户输入是否在前端进行了基本校验
- [ ] 文件上传是否校验了类型和大小
- [ ] 表单提交是否防止了重复提交
- [ ] 是否有 CSRF 防护措施

### 依赖安全
- [ ] 是否使用了已知存在漏洞的依赖版本
- [ ] 是否避免了从不可信来源加载脚本

---

## 5. 业务逻辑 (Business Logic)

### 边界条件
- [ ] 空数据/空列表是否有合理的展示（Empty State）
- [ ] 超长文本是否有截断/换行处理
- [ ] 极端数值（0、负数、超大数）是否正确处理
- [ ] 网络异常/超时是否有兜底处理

### 错误处理
- [ ] API 请求失败是否有错误提示
- [ ] 是否有全局错误边界（Error Boundary）
- [ ] 异步操作是否有 loading 状态
- [ ] 错误信息是否对用户友好（不暴露技术细节）

### 状态管理
- [ ] 状态变更是否可预测（避免隐式状态修改）
- [ ] 异步操作是否有竞态条件处理（如快速连续点击）
- [ ] 组件卸载时是否取消了进行中的请求
- [ ] 多组件共享的状态是否放在合适的位置

### 数据一致性
- [ ] 乐观更新失败后是否回滚了 UI
- [ ] 缓存数据是否在适当时机失效/刷新
- [ ] 表单数据是否在提交后正确清理

---

## 6. 样式 (Styling)

### CSS 规范
- [ ] 类名是否遵循项目约定的命名规范（BEM / camelCase / kebab-case）
- [ ] 是否避免了过深的选择器嵌套（建议不超过 3 层）
- [ ] 是否避免了 `!important`（除非有充分理由）
- [ ] 是否避免了内联样式（除非动态计算）

### 响应式设计
- [ ] 是否考虑了移动端适配
- [ ] 是否考虑了平板/大屏等不同断点
- [ ] 是否使用了相对单位（rem/em/vw/vh）而非固定像素
- [ ] 图片和媒体元素是否自适应

### 设计系统一致性
- [ ] 颜色是否使用了设计 token/变量（而非硬编码色值）
- [ ] 间距是否使用了统一的间距系统
- [ ] 字体大小/字重是否使用了设计系统的值
- [ ] 圆角/阴影是否使用了设计系统的值

### 未使用样式
- [ ] 是否有未使用的 CSS 类
- [ ] 是否有被覆盖但从未生效的样式
- [ ] 删除组件时是否同步清理了相关样式

---

## 严重级别判定指南

| 级别 | 判定标准 | 示例 |
|------|----------|------|
| **P0（紧急）** | 安全漏洞、数据丢失、生产崩溃、无障碍关键流程阻断 | XSS 漏洞、硬编码 API 密钥、关键流程无法操作 |
| **P1（重要）** | 用户可见回归、SSR/水合失败、API 契约错误、严重性能问题 | 页面白屏、表单提交失败、内存泄漏、列表卡顿 |
| **P2（建议）** | 可维护性问题、性能隐患、缺少测试、非关键无障碍问题 | 缺少错误处理、重复代码、缺少 key 属性 |
| **P3（优化）** | 代码风格、命名优化、注释补充、微小改进 | 命名不够清晰、缺少注释、样式可优化 |
