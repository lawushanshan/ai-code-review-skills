# Frontend Code Review

对前端代码进行结构化审查。首次使用时自动检测项目技术栈并生成审查配置模板，后续调用基于模板进行审查。

## Intent

当用户要求审查、检查、分析前端代码时触发此技能。支持以下文件类型：
- `.tsx`, `.ts`, `.jsx`, `.js`, `.vue`, `.svelte`
- `.css`, `.scss`, `.less`
- `.html`

## Review Modes

根据用户意图自动判断使用哪种模式。如果用户没有明确指定，**默认使用模式 B（Git 变更审查）**。

### Mode A: 项目全量审查

**触发条件**：用户明确说"审查整个项目"、"全量审查"、"检查所有代码"
**审查范围**：项目中所有前端文件（排除配置中的排除路径）
**适用场景**：项目初始化后的首次审查、定期代码健康检查

### Mode B: Git 变更审查（默认）

**触发条件**：用户说"审查我的修改"、"检查这次改动"、"提交前审查"、"review 一下"、或任何未明确指定范围的审查请求
**审查范围**：仅本次 git 变更涉及的文件和代码行
**适用场景**：日常开发中的小改动、提交前的快速检查

支持以下 git 范围（按优先级自动判断）：

| 优先级 | 范围 | Git 命令 | 说明 |
|--------|------|----------|------|
| 1 | Staged 变更 | `git diff --cached` | 已 `git add` 但未 commit 的变更 |
| 2 | Unstaged 变更 | `git diff` | 已修改但未 `git add` 的变更 |
| 3 | 最近一次 commit | `git diff HEAD~1 HEAD` | 最后一次提交的变更 |
| 4 | 当前分支 vs 主分支 | `git diff main...HEAD` 或 `git diff master...HEAD` | 当前分支所有变更 |

如果同时存在 staged 和 unstaged 变更，**优先审查 staged 变更**，并提示用户还有 unstaged 变更未审查。

### Mode C: 文件定向审查

**触发条件**：用户指定了具体文件或目录路径
**审查范围**：用户指定的文件或目录
**适用场景**：审查特定组件、模块或页面

## First-Time Setup: 技术栈检测与模板初始化

### 检测流程

首次调用时，检查项目根目录下是否存在配置文件 `.skills/frontend-code-review/.review-config.md`。

**如果不存在**，执行以下初始化流程：

1. **扫描项目根目录**，检测以下信息：

| 检测项 | 检测方式 |
|--------|----------|
| 框架 | 读取 `package.json` 的 dependencies：React/Vue/Angular/Svelte/Next.js/Nuxt 等 |
| 语言 | 检查是否使用 TypeScript（`tsconfig.json` 存在或 `.ts/.tsx` 文件占比） |
| CSS 方案 | 检测 Tailwind CSS / UnoCSS / CSS Modules / Styled-components / Sass 等 |
| 状态管理 | 检测 Redux/Zustand/Pinia/Vuex/Jotai/Recoil 等 |
| 测试框架 | 检测 Vitest/Jest/Cypress/Playwright/Testing Library 等 |
| 构建工具 | 检测 Vite/Webpack/Rspack/Turbopack 等 |
| UI 组件库 | 检测 Ant Design/Element Plus/Shadcn/MUI/Radix 等 |
| 代码规范 | 检测 ESLint/Prettier/Biome 配置 |
| 包管理器 | 检测 pnpm/yarn/npm/bun |

2. **生成配置文件** `.skills/frontend-code-review/.review-config.md`，内容如下：

```markdown
# 前端代码审查配置

> 此文件由 frontend-code-review skill 自动生成，可根据项目需要手动修改。

## 项目技术栈

- **框架**: [检测到的框架及版本]
- **语言**: [TypeScript/JavaScript]
- **CSS 方案**: [检测到的 CSS 方案]
- **状态管理**: [检测到的状态管理方案]
- **测试框架**: [检测到的测试框架]
- **构建工具**: [检测到的构建工具]
- **UI 组件库**: [检测到的 UI 组件库]
- **代码规范**: [检测到的规范工具]
- **包管理器**: [检测到的包管理器]

## 审查规则配置

### 启用的检查类别（取消勾选以禁用）

- [x] 代码质量（命名、类型安全、代码结构）
- [x] 性能（不必要的重渲染、内存泄漏、包体积）
- [x] 可访问性（语义化 HTML、ARIA、键盘导航）
- [x] 安全性（XSS、CSRF、敏感数据暴露）
- [x] 业务逻辑（边界条件、错误处理、状态一致性）
- [x] 样式（CSS 规范、响应式、设计系统一致性）
- [ ] 国际化（i18n 硬编码检查）

### 项目特定规则（可自定义添加）

<!-- 在此添加项目特有的审查规则 -->
<!-- 例如：
- 所有 API 请求必须使用 useQuery/useMutation 封装
- 组件 Props 必须定义 TypeScript interface
- 禁止使用 any 类型
-->

### 排除路径（不审查的目录/文件）

- `node_modules/`
- `dist/`
- `build/`
- `coverage/`
- `*.min.js`
- `*.test.*` / `*.spec.*`（除非明确要求审查测试）

## 审查严重级别定义

- **P0（紧急）**: 安全漏洞、数据丢失、生产崩溃、无障碍关键流程阻断
- **P1（重要）**: 用户可见回归、SSR/水合失败、API 契约错误、键盘/焦点行为异常
- **P2（建议）**: 可维护性问题、性能隐患、重复状态、缺少测试
- **P3（优化）**: 代码风格、命名优化、注释补充
```

3. **提示用户**：告知已检测到技术栈并生成了配置文件，用户可以手动编辑该文件来自定义审查规则。

### 后续使用

如果配置文件已存在，直接读取配置并基于其中的规则进行审查。

## Review Process

### Mode A: 项目全量审查

1. 扫描项目中所有前端文件（排除配置中的排除路径）
2. 按文件逐一检查
3. 生成全量报告

### Mode B: Git 变更审查（重点）

#### Step 1: 获取变更范围

```bash
# 1. 先检查是否有 staged 变更
git diff --cached --stat
git diff --cached

# 2. 如果没有 staged，检查 unstaged 变更
git diff --stat
git diff

# 3. 如果都没有，检查最近一次 commit
git diff HEAD~1 HEAD --stat
git diff HEAD~1 HEAD

# 4. 如果都没有，检查当前分支 vs 主分支
git diff main...HEAD --stat  # 或 master...HEAD
git diff main...HEAD
```

#### Step 2: 过滤前端文件

从变更文件列表中，只保留前端相关文件（`.tsx`, `.ts`, `.jsx`, `.js`, `.vue`, `.svelte`, `.css`, `.scss`, `.less`, `.html`），排除配置中的排除路径。

如果没有前端文件变更，提示用户"本次变更不涉及前端文件"。

#### Step 3: 生成业务级变更概述

在分析 diff 的基础上，用**自然语言**总结本次变更的业务含义。这是让开发者一眼看懂"这次改了什么"的关键。

**生成方法**：
1. 读取所有变更文件的 diff 内容
2. 理解每个文件变更的业务意图（新增/修改/删除了什么功能）
3. 用简洁的中文概述每条业务变更

**概述要求**：
- 用**业务语言**而非技术语言描述（"移除了删除按钮"而非"删除了 3 行 JSX"）
- 每条概述控制在一句话内
- 按影响范围从大到小排列
- 标注变更类型：新增 / 修改 / 移除 / 重构 / 修复

**示例**：
```
- [移除] 用户列表页的"删除用户"按钮及相关确认弹窗
- [新增] 登录页支持"记住密码"功能，使用 localStorage 持久化
- [修改] 商品详情页的价格展示从红色改为品牌色，字号从 16px 调整为 20px
- [重构] 将散落在各组件中的 API 请求统一迁移到 useQuery hooks
- [修复] 修复了购物车数量在并发场景下显示不一致的问题
```

#### Step 4: 分析变更摘要

生成技术层面的变更摘要，包括：
- 变更文件数量
- 新增/修改/删除的文件列表
- 变更行数统计（+N / -N）

#### Step 5: 逐文件审查变更

对每个变更文件：

1. **读取 diff 内容**：只关注变更的代码行（`+` 和 `-` 行）
2. **读取上下文**：读取变更行周围的相关代码（前后各 20 行），理解组件结构、状态管理、依赖关系
3. **检查变更部分**：根据配置中启用的检查类别，只审查变更的代码，不审查未变更的代码
4. **检查变更引入的问题**：
   - 新增代码是否引入了 bug
   - 修改代码是否破坏了原有逻辑
   - 删除代码是否影响了其他依赖
   - 变更是否与项目规范一致

#### Step 6: 生成变更级报告

### Mode C: 文件定向审查

1. 读取用户指定的文件
2. 按检查类别全量审查
3. 生成文件级报告

---

## Checklist

详细审查规则见 [references/checklist.md](references/checklist.md)。

按配置文件中启用的检查类别，逐一检查：

- **代码质量**：命名、类型安全、职责单一、重复代码、导入导出
- **性能**：重渲染、虚拟滚动、懒加载、内存泄漏、代码分割
- **可访问性**：语义化 HTML、ARIA、键盘导航、颜色对比度、表单 label
- **安全性**：XSS、输入校验、敏感信息暴露、硬编码密钥
- **业务逻辑**：边界条件、错误处理、竞态条件、状态一致性
- **样式**：CSS 规范、响应式、未使用样式、设计 token 一致性

## Rules

1. **Git 变更审查只审查变更的代码**，不审查未变更的部分，保持报告简洁
2. 但需要**读取上下文代码**来理解变更的影响范围
3. 审查结果应具体、可操作，不要给出模糊的建议
4. 每个问题都要指出具体的文件和行号，并展示相关的 diff 内容
5. 如果问题数量超过 10 个，先输出"10+ 个问题"，然后只展示前 10 个最严重的问题
6. 不要包含主观偏好，只关注客观的代码质量
7. 审查完成后，询问用户是否需要帮助修复问题
8. 如果用户没有明确指定审查模式，**默认使用 Git 变更审查**
9. 如果没有发现问题，使用 Template B-B 输出简洁的"无问题"报告
